# 04 — 踩坑预警 (Gotchas, Anti-Patterns & Clever Hacks)

---

## 一、最容易产生的误解

### 误解 1：「Timeline 就是 Unity 的 Timeline 组件」

**事实**：完全不是。项目里的 `TimelineModel` / `TimelineObj` / `TimelineManager` 是 **自己手写的帧事件调度器**。作者在 `GameTimeline.cs` 的注释中明确说明：

> *"注意：和 Unity 的 timeline 不是一个东西，这个概念出来的时候 Unity 都还没出来。"*

它更接近 Flash 动画时代的"关键帧"概念，或者 MMORPG 服务端常见的"时间线技能系统"。如果你在 Unity 文档里搜 Timeline 的用法来理解这个项目，方向完全错了。

---

### 误解 2：「碰撞检测用的是 Unity 的 Collider / Rigidbody」

**事实**：本项目 **没有使用 Unity 内置物理系统做逻辑碰撞**。所有碰撞判断都是手写的数学计算：

- **子弹 vs 角色**：`BulletManager.FixedUpdate()` 中的圆形 vs 圆形碰撞
  ```csharp
  Mathf.Pow(dis.x, 2) + Mathf.Pow(dis.z, 2) <= Mathf.Pow(bRadius + cRadius, 2)
  ```
- **AoE vs 角色/子弹**：`AoeManager.FixedUpdate()` 中调用 `Utils.InRange()`
- **移动碰撞（地形阻挡）**：`MapInfo.FixTargetPosition()` + `Utils.CircleHitRect()`

这是 ARPG / MMORPG 领域常见的做法：逻辑碰撞不依赖物理引擎，确保确定性和可控性。

---

### 误解 3：「ChaState 就是一个普通的数据容器」

**事实**：`ChaState` 不仅存储数据，它还是角色的 **"调控中心"**。它在 `FixedUpdate()` 中承担了大量职责：

- Buff 的生命周期管理（tick / remove）
- 属性重算（`AttrRecheck()`）
- 将"输入指令"翻译为"表现指令"分发给 `UnitMove` / `UnitRotate` / `UnitAnim`
- 强制位移（`forceMove`）的合并

如果你只把它当成 DTO（数据传输对象），会漏掉一半的理解。

---

### 误解 4：「DesignerScripts 只是辅助函数」

**事实**：`DesignerScripts` 命名空间里的代码是 **整个游戏的"游戏性"所在**。框架代码（Manager / State）从设计上是 **行为无关** 的——它们不知道"子弹命中敌人该干什么"。所有的"该干什么"都存在于 `DesignerScripts` 中。

换句话说：如果你把 `DesignerScripts` 全部删除，框架还能运行，但子弹命中不会造成伤害、Buff 不会产生效果、Timeline 不会做任何事情。

---

### 误解 5：「`DesingerTables` 是拼写错误应该是 `DesignerTables`」

**事实**：确实是拼写错误（`Desinger` → `Designer`）。但这个拼写贯穿了整个项目的命名空间和所有引用，不要尝试去修正它——除非你打算做全局重命名。在阅读代码时保持心理映射即可：

- `DesingerTables` = 策划填的配置表
- `DesignerScripts` = 策划写的脚本逻辑（注意这个命名空间拼写反而是对的）

## 二、反直觉的设计

### 反直觉 1：`SceneVariants` 中到处是 `GameObject.Find()`

```csharp
public static void CreateBullet(BulletLauncher bulletLauncher){
    GameObject.Find("GameManager").GetComponent<GameManager>().CreateBullet(bulletLauncher);
}
```

每次调用都在做字符串查找。这在生产项目中是性能大忌，但在这个 Demo 中是可接受的权衡——它避免了引入单例模式的复杂度，让代码保持最简。

> **如果作为学习项目来改进**：可以将 `GameManager` 的引用缓存为静态字段，在 `GameManager.Start()` 中赋值。

---

### 反直觉 2：`BulletManager` 和 `AoeManager` 每帧都 `FindGameObjectsWithTag`

```csharp
GameObject[] bullet = GameObject.FindGameObjectsWithTag("Bullet");
GameObject[] character = GameObject.FindGameObjectsWithTag("Character");
```

这在对象数量多时会有性能问题。生产项目通常会维护一个列表来管理活跃对象。但这里的好处是 **逻辑简单、不需要手动管理注册/注销**。

---

### 反直觉 3：Buff 的属性修改不是实时的，而是靠 `AttrRecheck()` 重算

当 Buff 被添加或移除时，`ChaState` 不是增量调整属性，而是 **清零后从头重算所有属性**：

```csharp
private void AttrRecheck(){
    _controlState.Origin();
    this._prop.Zero();
    // 重新扫描所有 Buff 的属性修改
    for (int i = 0; i < this.buffs.Count; i++){
        for (int j = 0; j < buffProp.Length; j++){
            buffProp[j] += buffs[i].model.propMod[j] * buffs[i].stack;
        }
        _controlState += buffs[i].model.stateMod;
    }
    this._prop = (this.baseProp + this.equipmentProp + this.buffProp[0]) * this.buffProp[1];
}
```

这种"全量重算"策略虽然看起来低效，但它 **杜绝了"属性残留"Bug** —— 不会出现"Buff 移除后攻击力比之前高了"的情况。这在 Buff 系统中是非常常见且推荐的做法。

---

### 反直觉 4：`ChaProperty` 的乘法 `*` 不是数学意义上的乘法

```csharp
public static ChaProperty operator *(ChaProperty a, ChaProperty b){
    return new ChaProperty(
        Mathf.RoundToInt(a.moveSpeed * (1.0000f + Mathf.Max(b.moveSpeed, -0.9999f))),
        // ...
    );
}
```

`a * b` 的真实语义是 **「a 受到 b 的百分比修正」**：`result = a * (1 + b)`。  
所以 `buffProp[1].moveSpeed = 0` 表示"不增不减"，`-0.5` 表示"减少 50%"，`1.0` 表示"翻倍"。  
这与直觉中的 `a * b` 完全不同，初次阅读时极易误解。

---

### 反直觉 5：子弹命中时记录的是 `propWhileCast`（快照），而非实时属性

```csharp
// BulletState.InitByBulletLauncher()
if (this.caster && caster.GetComponent<ChaState>()){
    this.propWhileCast = caster.GetComponent<ChaState>().property;
}
```

子弹在发射瞬间会"拍照"记录施法者的属性。之后无论施法者的属性如何变化（加 Buff、脱装备、死亡），子弹的伤害都基于发射时的快照计算。

这是仿照 WoW 等 MMORPG 的设计——"飞行中的火球不会因为法师后来喝了一瓶智力药水而变强"。

## 三、巧妙的 Hack 技巧

### Hack 1：用 `onCast` Buff 实现"没子弹就自动换弹"

技能 `fire` 附带一个被动 Buff `AutoCheckReload`，它的 `onCast` 回调会拦截所有技能释放：

```csharp
private static TimelineObj ReloadAmmo(BuffObj buff, SkillObj skill, TimelineObj timeline){
    ChaState cs = buff.carrier.GetComponent<ChaState>();
    return (cs.resource.Enough(skill.model.cost) == true) ? timeline : 
        new TimelineObj(DesingerTables.Timeline.data["skill_reload"], buff.carrier, new object[0]);
}
```

如果弹药够 → 正常释放技能。如果弹药不够 → **偷天换日**，把技能的 Timeline 替换成"换弹"的 Timeline。框架代码完全不需要知道"换弹"这件事的存在。

> **设计含金量**：这展示了 Buff 的 `onCast` 回调能力远超"被动增伤"——它能 **替换技能的效果本身**。同样的机制可以实现"变身后技能形态切换"、"充能完成后释放升级版技能"等复杂效果。

---

### Hack 2：用"角色 + Buff"模拟炸药桶

炸药桶不是一个独立的 Entity 类型，它就是一个 **ChaState 角色**：

- `side = -1`（独立阵营，谁都能打它）
- `tags = ["Barrel"]`
- 挂载 `ExplosionBarrel` Buff：
  - `stateMod = ChaControlState.stun`（被"眩晕"所以不会动）
  - `onBeHurt = OnlyTakeOneDirectDamage`（只受 1 点伤害，桶对桶 9999）
  - `onTick = BarrelDurationLose`（每 5 秒自损 1 点 HP）
  - `onBeKilled = BarrelExplosed`（死亡时在脚下创建爆炸 AoE）

> **设计含金量**：没有给"可破坏物"写一行专门的代码，完全复用了角色+Buff 系统。这意味着理论上你可以用同样的方式实现"可被攻击的防御塔"、"可摧毁的障碍物"、"可触发的陷阱"等。

---

### Hack 3：用 `ChargeGoBack` 实现蓄力技能

`TimelineGoTo` 结构定义了一个"蓄力跳转点"：

```csharp
public struct TimelineGoTo{
    public float atDuration;   // 到达这个时间点
    public float gotoDuration; // 跳回到这个时间点
}
```

当 Timeline 推进到 `atDuration` 并且 `ChaState.charging == true` 时，时间指针会跳回 `gotoDuration`，形成循环。当玩家松开按键（`charging = false`），循环打破，Timeline 继续向后执行。

虽然本 Demo 中实际上没有使用蓄力（`TimelineGoTo.Null` 使其生效条件为 `atDuration = float.MaxValue`），但机制已就绪。你可以很容易用它实现"蓄力射击"——蓄力期间循环播放蓄力动画，松手后发射。

---

### Hack 4：传送子弹的"两阶段技能"

技能 `teleportBullet` 实际上有 **两种行为**，由一个被动 Buff 动态决定：

1. **第一次按下**：发射一颗传送子弹（`skill_teleportBullet_fire`），子弹的 `onCreate` 回调 `RecordBullet` 把子弹引用存入 Buff 参数。
2. **第二次按下**：`onCast` 回调 `FireTeleportBullet` 检测到已有记录的子弹 → 返回 `skill_teleportBullet_tele` Timeline → 添加 `TeleportTo` Buff → Buff 的 `onRemoved` 把角色传送到子弹位置 → 移除子弹。

整个"再按一次传送"的逻辑没有用任何状态机，全靠 **Buff 参数 + onCast 拦截** 实现。

---

### Hack 5：float 取模用 `RoundToInt(x * 1000) % RoundToInt(y * 1000)` 代替

在 Buff 的 `onTick` 检测和 AoE 的 `onTick` 检测中都能看到这个模式：

```csharp
if (Mathf.RoundToInt(buffs[i].timeElapsed * 1000) % Mathf.RoundToInt(buffs[i].model.tickTime * 1000) == 0)
```

直接用 `float % float` 会因为浮点精度问题错过 tick，所以先转成毫秒级整数再取模。这是游戏开发中处理定时器的常见小技巧。

## 四、"历史包袱"速查

| 问题 | 位置 | 说明 |
|------|------|------|
| 命名空间拼写错误 | `DesingerTables` | 贯穿全项目，不影响运行但阅读时注意 |
| `GameObject.Find()` 滥用 | `SceneVariants` 全部方法 | Demo 可接受，生产项目应缓存引用 |
| `FindGameObjectsWithTag` 每帧调用 | `BulletManager` / `AoeManager` | 性能不理想，可用列表管理替代 |
| UI 血条绑定方式 | `GameManager.Start()` 第 42 行 | 作者自己都注释了 "别问我这是什么狗屎了" |
| `ChaState` 职责过多 | `ChaState.cs` (~590 行) | 集属性/Buff/技能/移动/动画于一体，可拆但 Demo 规模没必要 |
| 移速/行动速度公式硬编码 | `ChaState.moveSpeed` / `actionSpeed` | 作者注释"这个公式也可以通过给策划脚本接口获得" |
| `ChaResource.Enough` 不够灵活 | `Structs/Character/ChaProperty.cs` | 只能判断 >=，不支持更复杂的条件 |
